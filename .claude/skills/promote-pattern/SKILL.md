## Your Task

Implement a specific prompt pattern as an automation artifact and open a PR for review.

### Input

`$ARGUMENTS` contains:
- A description of the pattern to implement
- `--as <type>` specifying the mechanism: `rule`, `skill`, `hook`, or `agent`

If `--as` is not specified, use your judgment based on the decision tree:
- Instructions about behavior/style → `rule`
- Repeatable user-triggered workflows → `skill`
- Automatic checks/actions on events → `hook`
- Specialized roles/personas → `agent`

### Implementation by Type

#### `--as rule`

Create `.claude/rules/<topic>.md` where `<topic>` is a kebab-case name derived from the pattern.

Format:
```markdown
---
# Optional: restrict to certain paths
# paths: src/api/**/*.ts
---

[Concise instructions that Claude should follow.
Write in imperative mood. Be specific. One rule per line.]
```

Risk: **LOW** → Label: `auto-merge`

#### `--as skill`

Create `.claude/skills/<name>/SKILL.md` where `<name>` is a kebab-case name.

Format:
```markdown
---
name: <name>
description: <when Claude should use this or what it does>
argument-hint: "[optional args]"
# Set user-invocable: false if Claude should auto-invoke based on context
# user-invocable: false
---

[Full skill prompt with instructions, steps, and output format]
```

Risk: **MEDIUM** → Label: `needs-review`

#### `--as hook`

This requires TWO changes:

1. If the hook needs a script, create it at `.claude/prompt-tracking/hooks/<name>.sh`
2. Update `.claude/settings.json` to add the hook configuration

Hook config structure (add to the appropriate event array):
```json
{
  "matcher": "ToolName",
  "hooks": [
    {
      "type": "command",
      "command": "bash .claude/prompt-tracking/hooks/<name>.sh",
      "async": true,
      "timeout": 10
    }
  ]
}
```

When updating settings.json, preserve ALL existing configuration. Read the file first, then merge the new hook into the existing hooks object.

Risk: **HIGH** → Label: `needs-review`

#### `--as agent`

Create `.claude/agents/<name>.md`.

Format:
```markdown
---
name: <name>
description: <when to delegate to this agent>
model: sonnet
tools: [appropriate tool list]
---

[Agent system prompt with role, behavior, and output expectations]
```

Risk: **MEDIUM** → Label: `needs-review`

### Git Workflow

After creating the artifact:

1. **Create branch:** `pattern/<type>-<short-name>` (e.g., `pattern/rule-typescript-style`)
2. **Stage the new/modified files**
3. **Update patterns.json:**
   - Read the current file
   - Add or update the pattern entry:
     ```json
     {
       "id": "pattern-<timestamp>",
       "description": "<what the pattern is>",
       "classification": "<instruction|workflow|capability|role>",
       "mechanism": "<rule|skill|hook|agent>",
       "target_file": "<path to created artifact>",
       "status": "proposed",
       "occurrences": <N>,
       "first_seen": "<date>",
       "last_seen": "<date>",
       "sample_prompts": ["<prompt1>", "<prompt2>"],
       "pr_url": "<filled after PR creation>",
       "created_at": "<now>"
     }
     ```
4. **Commit:** `pattern: add <type> for <short description>`
5. **Push:** `git push -u origin pattern/<type>-<short-name>`
6. **Open PR:**
   ```
   gh pr create \
     --title "Pattern: <short description>" \
     --label "<auto-merge|needs-review>" \
     --body "## Detected Pattern

   **What:** <description of the repeated behavior>
   **Occurrences:** <N> times
   **Sample prompts:**
   - \"<prompt 1>\"
   - \"<prompt 2>\"

   ## Automation Created

   **Type:** <rule|skill|hook|agent>
   **File:** \`<target path>\`
   **How it works:** <1-2 sentence explanation of what this automation does>

   ## Effect

   Once merged, <explain what changes for the developer — e.g., 'Claude will automatically follow this convention without being told'>

   ---
   *Generated by the pattern tracking system*"
   ```
7. **Update patterns.json** with the PR URL
8. **Amend the commit** with the updated patterns.json
9. **Force push** the branch (safe — it's a new branch)

### Important Rules

- NEVER commit to main — always use a feature branch
- NEVER modify existing rules, skills, hooks, or agents unless explicitly told to
- Read settings.json before modifying it — preserve existing config
- If a pattern with the same description already exists in patterns.json with status `approved` or `implemented`, warn and abort
- If the pattern exists with status `rejected`, warn and ask for confirmation before re-proposing
- Keep generated artifacts concise — rules should be 5-20 lines, not essays
